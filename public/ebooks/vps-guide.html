دليلك الشامل لرفع موقعك على VPS

من كتابة الكود إلى إطلاق الموقع باستخدام النسخة الأحدث من Docker

مقدمة: ليه تقرأ الكتاب ده؟

لو إنت بتعمل مواقع أو تطبيقات، أكيد جربت ترفع شغلك على استضافات عادية (Shared Hosting) وحسيت إنها بطيئة أو مقيدة، أو جربت خدمات سحابية ولقيت الفاتورة بتسحب منك أرقام خيالية. هنا بييجي دور الـ VPS (Virtual Private Server).

ببساطة، الـ VPS (Virtual Machine) هو سيرفر افتراضي داخل سيرفر أكبر حقيقي باستخدام تقنية (Hypervisor). بيوفر لك ببيئة معزولة تماماً كأنك تملك السيرفر لوحدك، شغال 24 ساعة، وبيديك تحكم كامل (Root Access). في الكتاب ده، هنمشي خطوة بخطوة عشان ترفع أي بروجيكت مهما كان حجمه (سواء كان بـ Node.js، React، PHP، Python أو غيره) على سيرفرك الخاص، وبأحدث الطرق الاحترافية.

ماذا ستحتاج للبدء؟

قبل الدخول في الخطوات العملية، يُفضل أن يكون لديك:

  • حساب GitHub: لرفع كود مشروعك عليه.
  • مشروع برمجي جاهز: المشروع الذي ترغب في رفعه على السيرفر.
  • اسم نطاق (Domain): لربطه بموقعك لاحقاً (مثل yourdomain.com).

الفصل الأول: البداية وتجهيز السيرفر

أول حاجة محتاجينها هي إننا نأجر السيرفر. في شركات كتير ممتازة وأسعارها رخيصة بتبدأ من 4 أو 5 دولار في الشهر، وأشهرها DigitalOcean، أو Hetzner (ممتازة تقنياً)، أو Hostinger، أو Contabo.

إيه الفرق بين خطط الـ KVM (من KVM 1 لـ KVM 8)؟ ومين يختار إيه؟

لما تيجي تشتري السيرفر من أي شركة، هتلاقي الخطط متقسمة بأسماء زي (KVM 1, KVM 2, KVM 4, KVM 8). الـ KVM هو اختصار لـ Kernel-based Virtual Machine، وهو نوع التقنية المستخدمة لفصل السيرفر الافتراضي بتاعك عن باقي السيرفرات عشان تضمن عزل النظام الخاص بك تماماً، हालांकि الموارد نفسها (الرامات والمعالج) قد تكون مشتركة مع مستخدمين آخرين على نفس الهاردوير الأساسي في بعض الشركات.

كل ما الرقم زاد، كل ما كانت مساحة السيرفر الافتراضية أكبر (معالج أقوى، رامات أكتر، ومساحة تخزين أكبر). إليك دليلك لاختيار الخطة المناسبة لك:

  • خطة KVM 1: (غالباً بتيجي بـ 1 vCPU معالج، و 4GB رامات، و 50GB مساحة).
    مناسبة لمين؟ دي الخطة المثالية للمبتدئين. لو بترفع مشروع تخرجك، أو موقع شخصي (Portfolio)، أو مدونة صغيرة متوقعش زيارات ضخمة في البداية. تقدر تشغل عليها 3 لـ 5 مواقع صغيرة بسهولة لو استخدمت Docker.
  • خطة KVM 2: (غالباً بتيجي بـ 2 vCPU معالج، و 8GB رامات، و 100GB مساحة).
    مناسبة لمين؟ لو هترفع متجر إلكتروني صغير لمتوسط (زي منصة ببيع عليها منتجات)، أو تطبيق ويب بيستخدم قواعد بيانات بشكل مستمر. الخطة دي بتديك مساحة ممتازة للتنفس وسرعة استجابة أعلى وقادرة تستحمل ترافيك معقول جداً يومياً.
  • خطة KVM 4: (غالباً بتيجي بـ 4 vCPU معالج، و 16GB رامات، و 200GB مساحة).
    مناسبة لمين؟ للشركات الصغيرة والمتوسطة، أو لو عندك منصة تعليمية فيها كورسات وفيديوهات وناس كتير بتسجل دخول في نفس الوقت. وبرضو ممتازة جداً لو إنت مطور وعندك أكتر من 10 أو 15 موقع عملائك شغالين عليهم، فبتحطهم كلهم على سيرفر واحد عشان توفر.
  • خطة KVM 8 (فأعلى): (بتيجي بـ 8 vCPU معالج، و 32GB رامات، و 400GB مساحة).
    مناسبة لمين؟ للمشاريع الضخمة (Enterprise Level)، والمتاجر الإلكترونية الكبيرة جداً اللي عليها آلاف الزيارات والتفاعلات في نفس اللحظة. نادراً ما ستحتاجها كمبتديء.

الخلاصة للبدء: ابدأ دائماً بخطة KVM 1 أو KVM 2. ميزة الـ VPS إنك تقدر تعمل Upgrade و تعلي الإمكانيات في أي لحظة بضغطة زرار لما موقعك يكبر ويزيد عدد زواره، بدون ما موقعك يقع أو تضطر تنقل الداتا.

خطوات حجز السيرفر:

  1. بتعمل حساب على أي شركة استضافة (مثل DigitalOcean أو Hetzner أو Hostinger).
  2. بتختار قسم الـ VPS Hosting (أو Droplet) وتختار الخطة المناسبة (KVM 1 أو KVM 2 مثلاً).
  3. بتختار نظام التشغيل Ubuntu (الإصدار 22.04 أو الأحدث).
  4. بعد الدفع، الشركة هتديك رقم الـ IP بتاع السيرفر، واسم المستخدم (اللي هو غالباً root)، والباسورد (أو يطلب منك تحدده وقت الإنشاء).

إزاي تدخل على السيرفر بتاعك؟

افتح الـ Terminal (في الماك أو اللينكس) أو CMD/PowerShell (في الويندوز)، واكتب الأمر ده عشان تتصل بالسيرفر:

ssh root@your_server_ip

(استبدل your_server_ip برقم السيرفر بتاعك)

هيطلب منك الباسورد، اكتبه (مش هيظهر قدامك وإنت بتكتبه للسرية) ودوس Enter. مبروك، إنت كده جوه السيرفر بتاعك!

أول وأهم خطوة (تحديث النظام): بمجرد الدخول للسيرفر لأول مرة، يجب عليك دائماً تحديث الحزم لتجنب تعارض النسخ لاحقاً. اكتب هذا الأمر: 
apt update && apt upgrade -y

الفصل الثاني: التأمين الجذري للسيرفر (Security Hardening)

حتى الآن، موقعك يعمل بكفاءة، لكن السيرفر الخاص بك مكشوف أمنياً. لكي يكون عملك احترافياً حقاً ويقاوم هجمات الاختراق في بيئة العمل الحقيقية (Production)، يجب تنفيذ هذه الخطوات الأمنية الثلاث فوراً.

1. تفعيل الجدار الناري (UFW Firewall)

السيرفر الجديد تكون جميع منافذه (Ports) مفتوحة افتراضياً. يجب إغلاق كل شيء باستثناء المنافذ التي نحتاجها (SSH للتحكم، و 80/443 للموقع).

apt install ufw
ufw allow OpenSSH
ufw allow 'Nginx Full'
ufw enable

بمجرد تفعيل الـ UFW، السيرفر سيرفض أي طلبات خارجية على أي بورت آخر غير المصرح بها.

2. حماية الـ SSH (تعطيل Root & Fail2ban)

ترك الدخول متاحاً للمستخدم root بكلمة مرور يجعل سيرفرك هدفاً سهلاً لبرامج التخمين العشوائي (Brute-force). الحل الأمثل هو إعداد SSH Keys (إصدار مفتاح مشفر لجهازك فقط) وتعطيل الدخول بكلمات المرور تماماً من ملف /etc/ssh/sshd_config.

بالإضافة لذلك، يجب تسطيب برنامج Fail2ban، وهو حارس ذكي يقوم بحظر الـ IP الخاص بأي شخص يحاول إدخال كلمة مرور خاطئة عدة مرات متتالية:

apt install fail2ban -y
systemctl enable fail2ban
systemctl start fail2ban

الفصل الثالث: سر المحترفين (تسطيب Docker الحديث)

زمان كنا بندخل نسطب كل حاجة بإيدينا.. ننزل قاعدة البيانات، وننزل بيئة التشغيل، والدنيا كانت بتضرب مننا. دلوقتي إحنا بنستخدم Docker. الدوكر بياخد البروجيكت بتاعك بكل اللي بيحتاجه، ويحطه في "صندوق" مقفول (Container) وشغال لوحده.

عشان نشتغل باحترافية، هنسطب النسخة الرسمية الأحدث من Docker (مش النسخة القديمة بتاعت أوبونتو). انسخ الأوامر دي بالترتيب في الـ Terminal:

1. تحديث النظام وتنزيل الأدوات الأساسية:

apt-get update
apt-get install ca-certificates curl gnupg

2. إضافة مفتاح الأمان الخاص بـ Docker:

install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | gpg --dearmor -o /etc/apt/keyrings/docker.gpg
chmod a+r /etc/apt/keyrings/docker.gpg

3. إضافة مستودع Docker الرسمي للنظام:

echo \
  "deb [arch="$(dpkg --print-architecture)" signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu \
  "$(. /etc/os-release && echo "$VERSION_CODENAME")" stable" | \
  tee /etc/apt/sources.list.d/docker.list > /dev/null

4. تسطيب أحدث إصدار من Docker و Docker Compose V2:

apt-get update
apt-get install docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin

كده إنت نزلت أحدث نسخة، وتقدر تتأكد بكتابة أمر docker -v وأمر docker compose version.

الفصل الرابع: التطبيق العملي ورفع الكود

عشان نرفع الكود، لازم يكون البروجيكت بتاعك مرفوع على GitHub.

1. سحب الكود للسيرفر:

جوه السيرفر، اكتب الأمر ده عشان تنزل الكود بتاعك:

git clone https://github.com/yourusername/your-repo-name.git

2. الدخول لمجلد المشروع:

cd your-repo-name

3. تشغيل المشروع:

بافتراض إنك مجهز ملف docker-compose.yml (أو compose.yaml) في البروجيكت بتاعك.
(مثال مبسط لشكل ملف الإعدادات المناسب لبيئة العمل الحقيقية Production):

services:
  web:
    build: . # أو استخدام صورة جاهزة بمكانها مثل image: repo/app:latest
    restart: unless-stopped # لضمان عودة الموقع للعمل تلقائياً إذا أعيد تشغيل السيرفر
    ports:
      - "127.0.0.1:3000:3000" # الأهم: نقيد الوصول ليكون داخلياً فقط ليمر عبر Nginx
    # ... باقي الإعدادات
انتبه لثغرة خطيرة: كتابة المنافذ بالشكل التقليدي "3000:3000" في دوكر تجعل موقعك متاحاً لأي شخص عبر http://ip:3000 متخطياً حماية الـ Firewall و Nginx تماماً! استخدم دائماً 127.0.0.1 لمنع ذلك.

أمر واحد بس هيشغل الموقع كله في الخلفية (لاحظ إننا بنستخدم docker compose من غير شرطة في النسخة الحديثة):

docker compose up -d

حرف الـ -d معناه "Detached"، يعني البروجيكت هيفضل شغال حتى لو قفلت الـ Terminal.

الفصل الخامس: ربط الدومين وتأمين الموقع

دلوقتي الموقع شغال على رقم IP وبورت معين (مثلاً 3000). بس إحنا عايزين الناس تدخل عن طريق لينك واسم دومين احترافي زي yourdomain.com.

1. ربط الدومين:

ادخل على الشركة اللي اشتريت منها الدومين (زي Namecheap أو GoDaddy)، وروح لإعدادات الـ DNS. ضيف سجل جديد نوعه A Record، واكتب فيه رقم الـ IP بتاع السيرفر بتاعك.

ملاحظة هامة: انتشار تحديثات الدومين DNS Propagation قد يستغرق من بضع دقائق إلى 24 ساعة، فلا تقلق إن لم يعمل الموقع فوراً.

2. تسطيب Nginx (موظف الاستقبال):

ده البرنامج اللي بيستقبل الزوار من الدومين ويدخلهم على البورت بتاع موقعك.

apt-get install nginx

3. إعداد Nginx لموقعك:

هنعمل ملف إعدادات جديد:

nano /etc/nginx/sites-available/yourdomain.com

حط جواه الكود ده (وغير yourdomain.com لدومينك، و 3000 للبورت بتاعك):

server {
    listen 80;
    server_name yourdomain.com www.yourdomain.com;

    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
        
        # تمرير الـ IP الحقيقي للزوار إلى تطبيقك
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

احفظ الملف (Ctrl+X ثم Y ثم Enter)، وبعدين فعل الإعدادات واعمل ريستارت:

ln -s /etc/nginx/sites-available/yourdomain.com /etc/nginx/sites-enabled/
systemctl restart nginx

4. تأمين الموقع (SSL مجاني):

عشان موقعك يكون https وآمن:

apt-get install certbot python3-certbot-nginx
certbot --nginx -d yourdomain.com -d www.yourdomain.com

اتبع التعليمات البسيطة على الشاشة، ومبروك موقعك بقى آمن!

الفصل السادس: إزاي تحدث موقعك بعد أي تعديل؟

طبعاً بعد ما رفعت موقعك، هتحتاج تعمل تعديلات وتصلح أخطاء. التحديث مش هياخد منك ثواني.

1. على جهازك أنت:

ارفع التعديلات الجديدة على GitHub كالعادة:

git add .
git commit -m "update features"
git push

2. على السيرفر:

ادخل على السيرفر، روح لفولدر المشروع، واسحب التعديلات:

cd your-repo-name
git pull

3. تطبيق التحديث (السر في --build):

قول لـ Docker يبني النسخة الجديدة بالتعديلات ويشغلها مكان القديمة بدون ما الموقع يقع فترة طويلة:

docker compose up -d --build
ملحوظة هامة: إذا واجهت أي مشكلة أو لم يعمل الموقع بعد التحديث، يمكنك استخدام الأمر docker compose logs -f لعرض السجلات وتتبع الأخطاء بسهولة.

الفصل السابع: استضافة أكثر من موقع على نفس الـ VPS (وفر فلوسك)

أكبر ميزة للـ VPS إنك تقدر تستضيف عليه 5 أو 10 مواقع بنفس الـ 5 دولار! السر كله في "البورتات".

1. تغيير البورت للموقع الثاني:

الموقع الأول شغال على بورت 3000. الموقع التاني لما تسحبه للسيرفر، ادخل على ملف docker-compose.yml بتاعه، وخلي البورت الخارجي مختلف، مثلاً 4000:3000. وشغله بـ docker compose up -d.

2. توجيه الدومين الثاني:

اعمل A Record للدومين التاني (site2.com) يشاور على نفس الـ IP بتاع السيرفر.

3. إعداد Nginx للموقع الثاني:

اعمل ملف إعدادات جديد للموقع التاني زي ما عملنا في الفصل الرابع:

nano /etc/nginx/sites-available/site2.com

وحط نفس كود الـ Nginx، بس غير server_name لـ site2.com، وغير proxy_pass لـ http://localhost:4000 (البورت الجديد)، وتأكد من وجود رؤوس (Headers) الـ Real-IP.

فعل الملف واعمل ريستارت:

ln -s /etc/nginx/sites-available/site2.com /etc/nginx/sites-enabled/
systemctl restart nginx

وأخيراً، أمن الموقع التاني بالـ Certbot:

certbot --nginx -d site2.com -d www.site2.com

الفصل الثامن: ترقية بيئة العمل (CI/CD Pipelines)

في الفصول السابقة، شرحنا تحديث الموقع عن طريق الدخول للسيرفر وكتابة git pull يدوياً. هذه الطريقة تعمل، لكنها ليست الأفضل للمحترفين.

في الشركات الحقيقية، نستخدم ما يُسمى بـ CI/CD (التكامل والتسليم المستمر) مثل GitHub Actions أو GitLab CI. الفكرة هي أنك بمجرد أن تعمل git push للكود الخاص بك من جهازك، يقوم GitHub أو GitLab تلقائياً بالدخول لسيرفرك، سحب التحديثات، إعادة بناء الـ Docker Image، وإعادة تشغيل الموقع دون أي تدخل يدوي منك. ابحث عن (GitHub Actions Docker VPS) لتتعلم كيفية تطبيقها، ستوفر عليك ساعات من العمل اليدوي!

الفصل التاسع: النسخ الاحتياطي والمراقبة (Backups & Monitoring)

لا تدع أول خطأ برمجي أو اختراق يدمر مجهود شهور. السيرفر القوي يجب أن يكون مراقباً ولديه نسخ احتياطية.

1. النسخ الاحتياطي (Backups): يمكنك استخدام أداة مثل rsync مع cron (مجدول المهام في لينكس) لأخذ نسخة من قاعدة بياناتك وملفاتك يومياً ونقلها لسيرفر تخزين آخر (Object Storage)، أو استخدام خدمة الباك أب التلقائية التي توفرها شركة الاستضافة (بتكلفة بسيطة إضافية).

2. المراقبة (Monitoring): لكي لا تُفاجأ بأن موقعك توقف لأن الرامات امتلأت، يفضل المحترفون تسطيب أدوات مجانية مثل Prometheus & Grafana. هذه الأدوات ترسم لك رسوماً بيانية توضح استهلاك المعالج والرامات لحظة بلحظة، بل وترسل لك رسالة تليجرام أو إيميل إذا كان السيرفر يعاني من ضغط شديد.

الخاتمة

مبروك! إنت دلوقتي مش مجرد شخص بيكتب كود، إنت بقيت بتعرف تعمل Deployment باحترافية وتدير السيرفر بتاعك زي مهندسين الـ DevOps. استخدامك للـ VPS مع أحدث إصدارات Docker و Nginx هيضمنلك سرعة، أمان، واستقرار لمشاريعك، والأهم إنه هيوفر عليك ميزانية ضخمة.
